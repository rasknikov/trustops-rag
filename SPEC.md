# TrustOps RAG - Product Specification

Status: Draft v2.0  
Role: This document is the single source of truth for product scope, external contracts, business rules, and acceptance criteria. `ARCHITECTURE.md` explains how the system is built. `STACK.md` explains which technologies are used to build it.

## 1. Positioning

TrustOps RAG is a risk-aware enterprise RAG platform designed to answer only when evidence is sufficient, escalate uncertain or high-risk cases to human review, and preserve a full audit trail for governance.

It is not:

- a generic chatbot
- an autonomous agent that executes business actions
- a substitute for legal, compliance, finance, or policy owners

## 2. Product goals

The system must:

- answer with explicit citations to enterprise documents
- refuse or escalate when confidence or evidence is insufficient
- classify risk before final response routing
- measure confidence, cost, latency, and escalation quality
- remain reproducible in local mode and operable in production mode

## 3. Scope

### In scope

- question answering over approved enterprise documents
- document versioning and active-version retrieval
- risk-aware decisioning
- human-in-the-loop review workflow
- audit, metrics, and benchmark reporting

### Out of scope

- autonomous workflow execution
- internet search as part of the answer path
- direct mutation of source-of-record systems
- replacing formal human approval in regulated decisions

## 4. Actors

- End user: asks questions and receives answers, warnings, or escalation messages
- Reviewer: resolves tickets generated by the platform
- Content owner: publishes and activates document versions
- Platform operator: manages runtime, models, policies, and monitoring

## 5. Canonical business capabilities

1. Answer questions only from approved evidence.
2. Route high-risk or uncertain requests to human review.
3. Preserve traceability across question, retrieval, generation, validation, and decision.
4. Track operational cost and latency per request.
5. Learn from reviewer corrections without silently changing policy behavior.

## 6. Canonical status model

The product exposes five canonical response states.

| Status | Meaning | HTTP | Automatic answer allowed | Ticket allowed |
| --- | --- | --- | --- | --- |
| `answered` | Sufficient evidence and policy-approved response | `200 OK` | Yes | No |
| `answered_with_warning` | Partial evidence, still safe to return with explicit warning | `200 OK` | Yes | No |
| `human_review_required` | Business-safe refusal with ticket creation | `202 Accepted` | No | Yes |
| `blocked` | Request rejected due to security or policy violation | `403 Forbidden` | No | No |
| `failed_safe` | System could not produce a compliant answer and intentionally failed closed | `503 Service Unavailable` | No | Optional |

These statuses are mandatory across API, frontend, logs, dashboards, and tests.

## 7. API contract

### 7.1 POST `/ask`

#### Headers

- `Idempotency-Key` is optional but strongly recommended for retriable clients.
- The same `tenant_id` plus the same `Idempotency-Key` plus an equivalent normalized request payload must not create duplicate answers or duplicate tickets.
- `request_id` remains server-generated and is the authoritative trace identifier returned by the platform.

#### Request

```json
{
  "question": "Qual e o prazo de reembolso para compras internacionais?",
  "user_id": "user_123",
  "tenant_id": "tenant_demo",
  "metadata": {
    "department": "finance",
    "locale": "pt-BR",
    "channel": "web"
  }
}
```

#### Canonical response envelope

#### Field rules

- `request_id` is always present.
- `status` is always present and must be one of the five canonical states.
- `answer` is required only for `answered` and `answered_with_warning`.
- `message` is required for `human_review_required`, `blocked`, and `failed_safe`.
- `policy_version` is always present and identifies the decision policy used for thresholds, routing, and safety behavior.
- `system_state` is always present and must be `normal` or `degraded`.
- `sources` is mandatory and non-empty for every automatic answer.
- `ticket_id` is required for `human_review_required` and optional for `failed_safe`.
- `warnings` contains machine-readable codes such as `partial_evidence`, `degraded_mode`, `policy_limited_scope`, or `cost_guard_triggered`.
- `decision_reason` is mandatory, comes from a closed enum, and must remain stable for analytics within a given `policy_version`.

#### `decision_reason` enum

The allowed initial values are:

- `policy_auto_answer`
- `partial_evidence_allowed`
- `insufficient_evidence`
- `high_risk_insufficient_evidence`
- `contradictory_evidence`
- `validation_failed`
- `out_of_scope`
- `prompt_injection_detected`
- `acl_denied`
- `cost_guard_triggered`
- `degraded_mode_fallback`
- `critical_dependency_unavailable`

Any addition, removal, or semantic change in this enum requires a `policy_version` bump.

### 7.2 Ticket APIs

The ticket contract is intentionally action-based, not endpoint-per-button.

- `GET /tickets?status=pending_review`
- `GET /tickets/{ticket_id}`
- `POST /tickets/{ticket_id}/actions`

#### Action request

```json
{
  "reviewer_id": "reviewer_007",
  "action": "approve",
  "corrected_answer": null,
  "notes": "Resposta aderente a politica vigente."
}
```

#### Allowed actions

- `approve`
- `edit`
- `reject`
- `mark_missing_policy`
- `escalate_to_specialist`

## 8. Business rules

### 8.1 Risk classification

| Risk level | Typical domains | Default behavior |
| --- | --- | --- |
| `low` | FAQ, onboarding, generic guidance | Automatic answer allowed when evidence is sufficient |
| `medium` | Internal policy, technical support, operational procedure | Automatic answer allowed with stricter validation |
| `high` | Legal, contractual, finance, compliance, HR-sensitive | Stronger routing, higher threshold, likely human escalation |

### 8.2 Evidence sufficiency

Evidence is considered sufficient only when all conditions below hold:

- `evidence_score >= 0.75`
- `unsupported_claims_count == 0`
- `contradiction_detected == false`
- at least one active document version supports the answer

### 8.3 Decision policy

#### Automatic answer

- `confidence >= 0.80`
- `risk_level` is `low` or `medium`
- evidence is sufficient
- request is within cost and latency guardrails

#### Answer with warning

- `0.60 <= confidence < 0.80`
- `risk_level` is `low` or `medium`
- `contradiction_detected == false`
- response includes explicit warning code

#### Human review required

- `confidence < 0.60`
- or `risk_level == high` and `confidence < 0.90`
- or `unsupported_claims_count > 0`
- or `contradiction_detected == true`
- or question is out of scope
- or validation fails after generation

#### Blocked

- prompt injection or prompt exfiltration attempt
- malicious or abusive input
- unauthorized tenant or ACL violation

#### Failed safe

- critical dependency unavailable after fallback
- audit trail could not be persisted
- policy engine could not reach a deterministic decision

## 9. Policy versioning

- `policy_version` governs decision thresholds, decision-reason taxonomy, escalation rules, cache eligibility, and degraded-mode behavior.
- Every answer, ticket, audit event, and evaluation run must be stamped with `policy_version`.
- Any behavioral change that can alter a request outcome requires a `policy_version` bump.

## 10. Document and cache policy

- Only active document versions may be used in the answer path.
- Newer active versions take precedence over older ones.
- High-risk responses must not be served from semantic cache.
- A human-corrected answer invalidates cache entries derived from the same evidence set.
- Cache keys must include question fingerprint, active document versions, model, and `policy_version`.

## 11. Human review policy

### Ticket creation triggers

- high-risk question
- low confidence
- contradiction across sources
- insufficient evidence
- missing or outdated document
- explicit user request for review
- fail-safe path with recoverable context

### Ticket minimum payload

```json
{
  "ticket_id": "HITL-00042",
  "request_id": "req_003",
  "question": "Posso cancelar esse contrato sem multa?",
  "draft_answer": "Nao ha evidencia suficiente para afirmar.",
  "retrieved_sources": [
    "contract_v3.pdf",
    "cancellation_policy_2026.pdf"
  ],
  "policy_version": "v1.2.0",
  "system_state": "normal",
  "confidence": 0.42,
  "risk_level": "high",
  "reason": "high_risk_insufficient_evidence",
  "status": "pending_review",
  "assigned_team": "legal"
}
```

### Review outcomes

- approved answer may be returned to the original requester if the product flow supports callback or polling
- edited answer becomes feedback data and invalidates related cache entries
- rejected answer is recorded as failure taxonomy input

## 12. Acceptance criteria

The product definition is only considered met when all statements below are true:

1. Every automatic answer includes at least one citation.
2. No automatic answer is returned when evidence is insufficient.
3. High-risk questions never bypass the stricter confidence threshold.
4. Every request produces a stable `request_id`, `policy_version`, and auditable `decision_reason`.
5. Equivalent retried requests under the same `Idempotency-Key` do not create duplicate business outcomes.
6. Ticket creation is deterministic for equivalent inputs under the same `policy_version`.
7. Document version changes invalidate affected cache usage.
8. The benchmark can compare baseline RAG versus TrustOps RAG using the same dataset and pinned `policy_version`.

## 13. Product success metrics

- citation coverage for automatic answers: `100%`
- correct escalation rate: target `>= 95%` on curated evaluation set
- unsupported automatic answer rate: target `0%` on release gate dataset
- cost per query and escalation rate tracked by domain and risk level
- benchmark reports available for baseline and TrustOps pipelines
